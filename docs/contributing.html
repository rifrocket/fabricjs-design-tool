🚀 Quick Start for Contributors

Prerequisites

  • Node.js (v18 or higher)
  • npm (v9 or higher)
  • Git

Development Setup

1. Fork and Clone the Repository

# Fork the repository on GitHub, then clone your fork
git clone https://github.com/YOUR_USERNAME/fabricjs-design-tool.git
cd fabricjs-design-tool

# Add the original repository as upstream
git remote add upstream https://github.com/rifrocket/fabricjs-design-tool.git

2. Install Dependencies

npm install

3. Verify Setup

# Run linting to check code quality
npm run lint

# Build the library
npm run build:lib

# Start development server
npm run dev

🔧 Development Workflow

1. Create a Feature Branch

# Update your fork
git checkout main
git pull upstream main

# Create a new feature branch
git checkout -b feature/your-feature-name

2. Make Your Changes

  • Follow the existing code style and patterns
  • Add TypeScript types for new functions/components
  • Update documentation if needed
  • Test your changes thoroughly

3. Code Quality Checks

# Run linting
npm run lint

# Fix any linting issues
npm run lint:fix

# Build to ensure no errors
npm run build:lib

4. Commit Your Changes

# Stage your changes
git add .

# Commit with a descriptive message
git commit -m "feat: add new feature description"

# Push to your fork
git push origin feature/your-feature-name

5. Create a Pull Request

  1. Go to the original repository on GitHub
  2. Click "New Pull Request"
  3. Select your feature branch
  4. Fill out the PR template with details about your changes

📝 Code Style Guidelines

TypeScript

  • Use proper TypeScript types (avoid any when possible)
  • Export interfaces and types from appropriate files
  • Use meaningful variable and function names

React

  • Use functional components with hooks
  • Follow React best practices for hooks dependencies
  • Use proper prop types and component interfaces

Naming Conventions

  • Files: Use kebab-case (my-component.tsx)
  • Components: Use PascalCase (MyComponent)
  • Functions: Use camelCase (myFunction)
  • Constants: Use UPPER_SNAKE_CASE (MY_CONSTANT)

📦 Publishing (Maintainers Only)

⚠️ Note: Publishing is restricted to maintainers only. Contributors should focus on submitting pull requests.

Automated Publishing

The project uses GitHub Actions for automated publishing:

  1. Bump Version: npm version patch|minor|major
  2. Push to npm-production: git push origin npm-production
  3. Action Runs: Automatically builds, tests, and publishes

🤝 Pull Request Guidelines

Before Submitting

  • ☐ Code follows project style guidelines
  • ☐ All tests pass
  • ☐ Documentation is updated
  • ☐ Linting passes without errors
  • ☐ Build succeeds

PR Template

## Description
Brief description of changes

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Breaking change
- [ ] Documentation update

## Testing
- [ ] Added tests for new functionality
- [ ] All existing tests pass
- [ ] Manually tested changes

## Checklist
- [ ] Code follows style guidelines
- [ ] Self-review completed
- [ ] Documentation updated
- [ ] No new warnings introduced
